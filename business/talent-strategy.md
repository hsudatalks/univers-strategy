# 人才战略

> 基于对 AI 局限性的深刻理解，找准人类的真实价值，组织正确的人才来充分发挥 AI 带来的价值。

## 核心理念

### AI 时代的人才悖论

**传统认知**：
- AI 越强 → 人类价值越低 → 焦虑

**我们的认知**：
- AI 越强 → 更需要善于选择尺度和框架的人 → 机会

**关键洞察**：
AI 的缩放能力有根本限制（见 [PHILOSOPHY.md](../PHILOSOPHY.md)），因此**善于上下文自由缩放的人**成为稀缺资源。

## 我们要什么样的人？

### 一、核心特质：缩放者（Scalers）

不是"AI 专家"，不是"领域专家"，而是**能在多个尺度间自由切换的思考者**。

#### 特质 1：多尺度思维

**表现**：
- 能在细节和全局间自由切换，不会困在单一层级
- 讨论具体问题时，会主动上升到原则层
- 讨论战略时，会下沉到具体操作验证

**反例**：
- ❌ "这个技术方案有XX优点"（只在技术尺度）
- ✅ "这个技术方案短期快，但长期看会增加维护成本，从用户角度看..."

**识别方法**：

**面试题 1：决策的多尺度分析**
```
情境：你最近做了一个重要决策（技术、产品、生活都可以）

问题：
1. 当时是怎么想的？
2. 现在回看，从不同时间尺度（一周后、一月后、一年后）看结果如何？
3. 如果让你重新决策，会考虑哪些当时没考虑的维度？

评判：
- 优秀：能主动从多个时间尺度分析，发现当时的局限
- 中等：能回答，但需要引导
- 不合格：只能从单一尺度看，或没有反思
```

**面试题 2：切换视角**
```
情境：给一个产品功能设计

问题：
- 从用户角度：这个功能解决什么问题？
- 从工程角度：实现难度和风险是什么？
- 从商业角度：这对公司价值是什么？
- 从长期角度：这会不会引入新问题？

评判：
- 优秀：能流畅切换，发现不同视角的冲突并思考权衡
- 中等：能切换但不深入
- 不合格：只能从自己的专业视角看
```

#### 特质 2：框架质疑能力

**表现**：
- 不满足于"在框架内优化"
- 会问："我们解决的问题本身对吗？"
- 遇到困难时，能重新定义问题

**反例**：
- ❌ "这个优化目标很难达到，我们需要更好的算法"
- ✅ "等等，这个优化目标本身合理吗？我们是否在优化错误的东西？"

**识别方法**：

**面试题 3：框架质疑**
```
情境：公司要优化"客户响应时间"，从平均2小时降到30分钟

问题：
1. 你会怎么做这个优化？
2. （等候选人回答后）你觉得这个优化目标本身合理吗？
3. 有没有可能我们在优化错误的指标？

评判：
- 优秀：能质疑目标（例如："也许客户在乎的不是速度，而是确定性"）
- 中等：只在引导下才能质疑
- 不合格：完全接受目标，只想怎么实现
```

**面试题 4：重新定义问题**
```
情境：用户投诉"系统太慢"

问题：
- 大部分人会想："怎么优化性能？"
- 你会怎么处理？

评判：
- 优秀：会质疑"慢"的定义，探索真实问题（可能是等待焦虑，而非绝对速度）
- 中等：会优化性能，但也会调研用户需求
- 不合格：直接开始性能优化
```

#### 特质 3：整合性认知

**表现**：
- 能整合技术、业务、用户、伦理等多个维度
- 不会说"这不是我的领域"
- 能理解不同利益相关者的需求

**识别方法**：

**面试题 5：多维度决策**
```
情境：AI 优化建议关闭部分区域空调以节能20%，但可能影响10%用户舒适度

问题：
- 作为技术负责人，你怎么决策？
- 如果你是产品负责人/客户/CEO，分别会怎么想？
- 这几个视角有冲突吗？如何权衡？

评判：
- 优秀：能理解多方视角，提出平衡方案，甚至重新定义问题
- 中等：理解冲突但难以权衡
- 不合格：只从技术角度看（"优化效果好，应该推行"）
```

#### 特质 4：元认知能力

**表现**：
- 能监控和调节自己的思考过程
- 知道"我现在在什么层级思考"
- 能主动调整思考策略

**识别方法**：

**面试题 6：思考过程的反思**
```
情境：让候选人现场解决一个问题（编码或设计）

问题（解决后）：
- 你的思考过程是怎样的？
- 中间有改变想法吗？为什么？
- 回看整个过程，有哪里可以做得更好？

评判：
- 优秀：能清晰描述思考过程，识别自己的假设和调整
- 中等：能描述过程但不深入
- 不合格：说不清楚怎么想的，或认为没什么可改进
```

### 二、具体角色画像

#### 角色 1：人机协作设计师

**不是传统 UX 设计师**，而是理解认知科学和 AI 局限的交互专家。

**核心能力**：
1. **理解人类的缩放能力**
   - 学过认知科学、心理学、或有相关直觉
   - 能设计支持多尺度思考的界面

2. **理解 AI 的局限**
   - 知道 AI 不能做什么
   - 能设计人机分工的边界

3. **原型快速验证**
   - 不只是画图，要能验证"这个设计真的帮助人类缩放了吗"

**招聘来源**：
- 认知科学 + 设计背景
- HCI（人机交互）研究者
- 有深刻用户洞察的产品设计师

**面试重点**：
- 给一个人机协作场景，设计界面
- 评判是否真正支持人类的优势（而非只是"好看"）

#### 角色 2：Skills 架构师

**不是传统后端架构师**，而是理解 AI 调用模式的 API 设计专家。

**核心能力**：
1. **理解 AI 的认知模式**
   - 什么样的 API AI 容易理解？
   - 如何让 Skills 自描述？

2. **抽象和封装能力**
   - 把复杂的 Workbench 能力封装成简单 API
   - 平衡能力和复杂度

3. **演进设计**
   - Skills 要能平滑升级
   - 保持向后兼容

**招聘来源**：
- 有 API 设计经验的后端架构师
- 做过 AI 工具/平台的工程师
- 开源项目的维护者

**面试重点**：
- 给一个复杂功能，设计 Skills API
- 评判是否简单、清晰、AI 友好

#### 角色 3：业务 + AI 融合专家

**不是"业务提需求，AI 实现"的割裂模式**，而是既懂行业又懂 AI 的复合人才。

**核心能力**：
1. **深度行业知识**（如 HVAC）
   - 理解业务逻辑、痛点、机会

2. **AI 应用能力**
   - 不需要从零训练模型
   - 但要知道 AI 能做什么、不能做什么

3. **缩放能力**
   - 能在业务尺度和技术尺度间切换
   - 能把业务问题转化为 AI 可解决的问题

**招聘来源**：
- 行业专家（如 HVAC 工程师）+ AI 培训
- AI 工程师 + 行业深入学习
- 最好：两者兼具的跨界人才

**面试重点**：
- 给一个行业问题，问如何用 AI 解决
- 评判是否理解业务本质，而非生搬硬套 AI

## 组织结构

### 传统 AI 公司的问题

```
传统结构：
├── AI 团队 → 做模型
├── 工程团队 → 做系统
├── 产品团队 → 做界面
└── 业务团队 → 提需求

问题：
1. 没人负责"人机协作"
2. 团队间严重割裂
3. AI 和业务脱节
4. 没人从元层思考"我们在做对的事吗"
```

### Univers 的结构

```
Univers 结构：

纵向（按三层架构）：
├── Workbench 团队
│   ├── 人机协作设计师（lead）
│   ├── 前端工程师
│   └── 后端工程师
├── Skills 团队
│   ├── Skills 架构师（lead）
│   ├── AI 工程师
│   └── API 设计师
└── Operation 团队
    ├── HVAC 行业组
    │   ├── 业务 + AI 专家（lead）
    │   └── AI 工程师
    ├── 照明行业组
    └── ...

横向（跨团队研究）：
└── 缩放能力研究组
    ├── 成员来自各团队
    ├── 研究：如何更好地支持人类缩放
    └── 输出：设计原则、最佳实践

治理层：
└── 元层委员会
    ├── 定期审视：我们在做对的事吗？
    ├── 质疑框架和假设
    └── 战略方向调整
```

### 关键机制

**1. 缩放能力研究组**
- **目标**：持续研究如何更好地支持人类的多尺度思考
- **成员**：来自各团队的"缩放能力强"的人（不按职级）
- **产出**：
  - 每季度发布"缩放能力报告"
  - 提出产品和技术改进建议
  - 培训材料

**2. 元层委员会**
- **目标**：跳出日常工作，从元层审视公司方向
- **成员**：CEO + 各团队 lead + 轮值成员
- **频率**：每月一次
- **议题**：
  - 我们最近在优化什么？这个目标对吗？
  - 从长期看，我们的方向有偏离吗？
  - 有没有被忽视的重要维度？

**3. 跨团队轮岗**
- 让工程师做产品，产品做工程
- 培养多视角思维
- 每年至少一次

## 招聘策略

### JD 示例：人机协作设计师

```markdown
# 人机协作设计师

## 我们在找什么样的人？

不是传统 UX 设计师，而是理解人类认知、AI 局限、能设计真正增强人类能力的交互专家。

## 你会做什么？

- 设计 Workbench 的多尺度视图系统
- 研究：什么样的界面真正帮助用户在不同尺度间切换？
- 与 AI 工程师合作，设计人机分工边界
- 验证：用户的掌控感是否提升？

## 我们看重的能力：

**核心能力（必须）**：
1. **多尺度思维**：能从细节、策略、目标等多个层级思考
2. **认知理解**：理解人类如何思考、决策、感知
3. **原型验证**：能快速做出原型并验证假设

**加分项**：
- 认知科学、心理学背景
- 做过 AI 相关产品
- HCI 研究经验

**我们不看重**：
- 传统设计工具的熟练度（可以学）
- 作品集的"好看"程度（更看重思考深度）

## 面试流程：

1. 初筛：聊聊你对人机协作的理解
2. 设计任务：给一个多尺度思考的场景，设计界面
3. 原型评审：你如何验证这个设计真的有用？
4. 文化匹配：聊聊你的思考方式

## 我们提供：

- 重新定义人机协作的机会
- 跨学科的团队（认知科学 + AI + 工程）
- 每季度的"缩放能力研究"时间
- 开放的氛围：鼓励质疑和重新定义问题
```

### 面试流程设计

**传统面试的问题**：
- 只测技能（算法题、系统设计）
- 不测思维方式

**Univers 面试流程**：

#### 第一轮：缩放能力测试（1小时）

**目标**：评估多尺度思维和框架质疑能力

**题目类型**：
1. **决策分析**："最近做的一个决策，从多个时间尺度看"
2. **框架质疑**："这个优化目标合理吗？"
3. **视角切换**："从不同角色看这个问题"

**评分**：
- 能主动缩放：+2
- 需要引导才能缩放：+1
- 只能单一尺度：0
- 每道题独立评分，总分 ≥ 4 分进入下一轮

#### 第二轮：实际任务（2小时）

**根据角色不同**：
- 设计师：设计多尺度视图
- 工程师：设计 Skills API
- 业务专家：AI 如何解决行业问题

**评判标准**：
- 是否展现缩放能力？
- 是否质疑过假设？
- 技术/设计能力如何？

#### 第三轮：文化匹配（30分钟）

**目标**：判断是否认同理念

**问题**：
- 你怎么看 AI 和人类的关系？
- 你认为人类在 AI 时代的价值是什么？
- 你更喜欢"完成任务"还是"重新定义问题"？

**红旗**：
- 盲目崇拜技术
- 认为 AI 会替代一切
- 只想"执行"，不想"思考"

## 文化建设

### 核心价值观

**1. 尊重人类智慧**
- AI 是工具，人类是主体
- 技术服务于目标，不是炫技

**实践**：
- 每次 demo 强调："这帮助用户更好决策了吗？"
- 禁止"AI 会取代XX"的言论
- 鼓励质疑 AI 的建议

**2. 鼓励多尺度思考**
- 不满足于完成任务
- 定期反思：方向对吗？

**实践**：
- **每月缩放会议**：所有项目从元层审视
- **最佳框架质疑奖**：奖励重新定义问题的人
- **轮岗制度**：体验不同视角

**3. 追求真正价值**
- 不做表面功能
- 追问：这对用户的真正价值是什么？

**实践**：
- 产品评审必问："这解决了什么真实问题？"
- 技术选型必问："这对用户有什么价值？"
- OKR 设置：不只是数字，要有意义

### 具体机制

**1. 每月缩放会议**
- **时间**：每月第一个周五下午
- **参与**：全员
- **流程**：
  1. 回顾本月所有项目
  2. 从元层审视：我们在优化什么？目标对吗？
  3. 从长期看：方向有偏离吗？
  4. 从用户角度：我们忽视了什么？
- **产出**：可能调整方向、重新定义问题

**2. 最佳框架质疑奖**
- **频率**：每季度
- **标准**：重新定义问题，带来更好的解决方案
- **奖励**：不只是奖金，更是全公司分享

**3. 缩放能力工作坊**
- **频率**：每季度
- **内容**：
  - 认知科学：人类如何思考？
  - 哲学：什么是好的问题？
  - 案例分析：伟大的重新定义
- **讲师**：内部 + 外部专家

## 考核体系

### 不只看产出，更看思维方式

**传统考核问题**：
- KPI：功能数量、Bug 数量
- 只看结果，不看过程
- 鼓励"埋头做"，不鼓励"抬头看"

**Univers 考核**：

#### 个人层面

**1. 缩放能力展现（30%）**
- 是否在多个尺度上思考问题？
- 案例：
  - ✓ "我发现我们一直在优化X，但从长期看应该做Y"
  - ✓ "这个技术决策虽然快，但从用户角度可能有问题"

**2. 框架质疑（20%）**
- 是否质疑过假设和框架？
- 案例：
  - ✓ "这个功能需求，我们重新定义了问题"
  - ✓ "原来的目标不合理，我们调整了方向"

**3. 协作贡献（20%）**
- 是否帮助团队更好决策？
- 是否分享缩放思维？

**4. 专业能力（30%）**
- 技术/设计/业务能力
- 但不是唯一标准

#### 团队层面

**产品指标**：
- 用户掌控感评分（> 4.0/5.0）
- 用户愿意使用自动模式的比例（> 60%）
- 用户主动切换尺度的频率

**技术指标**：
- AI 在单一尺度上的优化效果
- Skills 易用性（AI 调用成功率 > 95%）
- 系统对多尺度思考的支持

**元层指标**：
- 每季度重新定义问题的次数
- 方向调整的及时性
- 长期目标的达成度

## 薪酬和激励

### 原则

**不只是市场价**，更要**奖励稀缺价值**：

| 能力 | 市场稀缺度 | Univers 重视度 | 薪酬倍数 |
|------|----------|---------------|---------|
| AI 算法调参 | 中 | 低 | 1.0x |
| 传统工程能力 | 中 | 中 | 1.2x |
| **缩放思维 + 技术** | **高** | **极高** | **1.5-2.0x** |
| **重新定义问题的能力** | **极高** | **极高** | **2.0-3.0x** |

### 晋升标准

**传统公司**：
- P5: 独立完成任务
- P6: 带小项目
- P7: 带大项目
- P8: 技术专家

**Univers**：
- **L1（执行者）**: 完成任务
- **L2（缩放者）**: 能在多个尺度思考，质疑假设
- **L3（架构师）**: 设计支持缩放的系统
- **L4（战略家）**: 从元层定义方向

**重点**：L2 是关键分水岭，不是所有人都能晋升到 L2。
