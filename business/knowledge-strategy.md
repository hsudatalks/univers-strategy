# 垂直知识战略

## 执行摘要

在 AI 时代，**垂直领域知识的价值正在重构**。本文档定义 Univers 在垂直知识领域的战略定位和商业模式。

**核心洞察**：
- **可编码知识**：AI 优势，人类价值下降 → 让 AI 做
- **情境知识**：AI 困难，人类短期优势 → 人机协作
- **元层知识**：AI 几乎没有，人类核心价值 → 人类主导，AI 执行

**战略方向**：
1. 不要试图"教会 AI 所有领域知识"（死路）
2. 构建"元层知识驱动，AI 执行"的商业模式（活路）
3. 让领域专家专注于"什么是好问题"，而非"怎么解决问题"

**商业模式**：从"卖工具"到"卖服务" → AI 驱动的专业服务，按价值收费

---

## 一、垂直知识的三个层次

### 1.1 可编码知识（Codifiable Knowledge）

#### 定义
可以明确表达、形式化的知识：规则、公式、流程、标准。

#### 示例（HVAC 领域）
```yaml
供暖系统参数:
  温度范围: 18-24°C
  湿度范围: 40-60%

能效公式:
  COP = 制热量 / 耗电量

标准流程:
  1. 检查传感器
  2. 校准基准线
  3. 调整控制参数
  4. 记录日志
```

#### AI 能力评估
- **学习难度**：✅ 容易
- **执行质量**：✅✅ 比人更好（不会出错、不会遗漏）
- **泛化能力**：✅ 容易迁移到其他场景

#### 人类价值变化
**⬇️ 相对下降**：AI 更擅长记忆、计算、执行规则

**战略含义**：
- ❌ 不要让人去记、去算、去执行规则
- ✅ 让 AI 做，释放人力
- ✅ 但需要人类判断"何时应用哪个规则"

---

### 1.2 情境知识（Contextual Knowledge）

#### 定义
依赖具体情境、难以明确表达的知识：经验模式、直觉判断、局部诀窍。

#### 示例（HVAC 领域）
```
场景 1：老师傅的经验
"这个压缩机的声音不对，虽然参数正常，但肯定快坏了"

场景 2：客户经理的判断
"这个客户表面上在乎能耗，其实更在乎设备维护频率"

场景 3：技师的诀窍
"这个区域冬天经常有凝露，虽然湿度显示正常，
但实际需要提前 2 小时预热"
```

#### AI 能力评估
- **学习难度**：⚠️ 困难（需要大量标注数据，且难以泛化）
- **执行质量**：⚠️ 不稳定（在训练分布内可以，分布外不行）
- **泛化能力**：❌ 差（"这个声音不对"很难迁移到新设备）

#### 人类价值变化
**📈 短期上升**：人类的经验、直觉、对情境的感知仍然重要

**但长期存在风险**：
- 如果这类知识可以被大量标注、被结构化 → AI 最终会学会
- 这类知识的价值"半衰期"在缩短

**战略含义**：
- ✅ 短期：人机协作（人类提供情境判断，AI 执行）
- ⚠️ 中期：将情境知识"显性化"（转化为可编码知识）
- ❌ 长期：不能完全依赖情境知识作为护城河

---

### 1.3 元层知识（Meta-level Knowledge）

#### 定义
关于"什么是好问题"、"什么真正重要"的知识：价值判断、目标设定、框架质疑。

#### 示例（HVAC 领域）
```
错误的问题：
"如何让这个空调更省电 5%？"

更好的问题：
"客户真正在乎的是什么？"
→ 发现：客户其实在乎室内空气质量，而非能耗
→ 重新定义问题：如何在保证空气质量前提下优化能耗？

元层洞察：
"HVAC 行业的真正痛点不是能效，而是故障预测"
→ 这个洞察来自对行业、客户、成本结构的深刻理解
→ AI 无法主动得出，因为 AI 不知道"什么是痛点"
```

#### AI 能力评估
- **学习难度**：❌ 几乎不可能（需要"主动质疑框架"的能力）
- **执行质量**：❌ 没有（AI 只能在给定框架内优化）
- **泛化能力**：❌ 无（AI 不会自发地"跳出框架"）

#### 人类价值变化
**📈📈 大幅上升**：这是人类在 AI 时代的核心价值

**为什么 AI 做不到**：
1. AI 没有"内在动机" → 不知道什么"真正重要"
2. AI 没有"上下文自由缩放" → 不会主动切换尺度
3. AI 没有"质疑框架"的能力 → 不会问"我们解决的是对的问题吗？"

**战略含义**：
- ✅✅✅ 这是我们的护城河
- ✅✅✅ 让领域专家专注于元层思考
- ✅✅✅ 构建"元层驱动，AI 执行"的商业模式

## 二、Univers 的战略定位

### 2.1 核心定位

**我们不是**：
- ❌ "HVAC 领域的 SaaS 平台公司"（卖软件订阅）
- ❌ "帮客户省电的公司"（卖可编码知识）
- ❌ "替代老师傅的公司"（卖情境知识）

**我们是**：
- ✅ "AI 驱动的 HVAC 优化服务公司"（卖结果）
- ✅ "让领域专家专注于'什么是好问题'，AI 负责'怎么解决问题'"
- ✅ "构建元层知识驱动、AI 执行的专业服务"

### 2.2 价值主张

#### 对领域专家
```
之前：
你 70% 时间在做 AI 能做的事（执行规则、计算、监控）
你 30% 时间在做你真正擅长的事（发现问题、判断价值、提出洞察）

现在（使用 Univers）：
你 90% 时间在做你真正擅长的事
你 10% 时间在引导 AI（告诉它"什么重要"）
```

**具体场景**：
```
老方式：
1. 老师傅每天巡检 50 台设备（8 小时）
2. 发现 1 台设备"声音不对"（凭经验）
3. 手动记录、分析、出报告（2 小时）
4. 留给"思考改进方案"的时间：0 小时

Univers 方式：
1. AI 自动巡检、监控、记录（24/7）
2. AI 发现异常，推送给老师傅
3. 老师傅判断："这是问题吗？" → 5 分钟
4. 老师傅思考："为什么老出这类问题？根本原因是什么？" → 7 小时
5. 老师傅提出洞察："我们应该改变预防性维护策略"
6. AI 执行新策略
```

#### 对企业客户
```
之前：
雇佣 10 个工程师 → 9 个在执行，1 个在思考
总成本：10 × 工资
创新能力：1 人份

现在（使用 Univers）：
雇佣 3 个工程师 + Univers 平台
→ 3 个都在思考，AI 负责执行
总成本：3 × 工资 + 平台费用
创新能力：3 人份
```

### 2.3 与竞争对手的差异

#### 对比：传统 HVAC 软件
```
传统软件：
定位：SaaS 平台（卖工具订阅）
核心：执行可编码知识（监控、控制、优化）
用户角色：操作员
价值：降低人力成本
客户做：学习使用软件

Univers：
定位：AI 驱动的专业服务（卖结果）
核心：元层洞察 + AI 执行
用户角色：领域专家
价值：提升创新能力
客户做：提需求、验收结果
```

#### 对比：通用 AI 平台（如 ChatGPT）
```
通用 AI：
能力：强大的可编码知识、一定的情境理解
局限：
- 缺乏领域纵深
- 无法主动发现"什么是好问题"
- 用户需要自己构建 prompt、自己做元层思考

Univers：
能力：通用 AI + 垂直领域元层知识
差异：
- 内置领域"好问题"库（策划好的 prompt）
- 自动提醒用户"被忽视的尺度"
- 沉淀用户的元层洞察，形成知识资产
```

## 三、商业模式设计

### 3.1 商业模式：服务为主

> **重要**：基于 ADR 004 和"SaaS 已死"趋势，Univers 定位为**AI 驱动的专业服务公司**，而非 SaaS 平台。

#### 核心模式：按价值收费的专业服务
```yaml
定位: AI 驱动的 HVAC 优化服务
客户购买: 结果（故障预测、能效优化、战略洞察）
客户体验: 提需求 → 验收结果
Univers 做: 用 AI + 元层知识为客户创造价值
收费方式: 按价值/结果收费

示例：
- 客户 A：故障率降低 30% → 收费故障成本节省的 20%
- 客户 B：能效提升 15% → 收费能源成本节省的 25%
- 客户 C：发现战略洞察 → 项目费 $50K-500K

关键: 客户不需要学习工具，直接买结果
```

#### Workbench 的定位
```yaml
主要定位（90%）: 内部运营平台
  - Univers 团队用来管理 AI 执行者
  - 提高服务效率和质量
  - 不对外销售

次要定位（10%）: 专家社区工具（可选）
  - 开放给极少数高认知专家
  - 建立专家网络和口碑
  - 免费或象征性收费
  - 不是主要收入来源

类比:
  - Uber 的调度系统（内部工具，规模化服务）
  - McKinsey 的知识库（内部工具，提升咨询质量）
```

#### 阶段 1：服务验证（Year 1）
```yaml
目标: 证明价值，建立口碑
方式: 我们管 AI，为客户交付结果

服务类型:
  - 故障预测服务
  - 能效优化服务
  - 战略洞察服务

定价:
  - 项目费: $10K-100K
  - 或按价值分成: 20-25%

客户数: 10-20 个标杆客户
收入目标: $500K-1M ARR

关键: 快速验证市场，打磨内部工具
```

#### 阶段 2：服务规模化（Year 2-3）
```yaml
目标: 规模化服务能力

分层服务:
  Tier 1（60-70%收入）: 高端服务
    - 我们管 AI，定制化交付
    - $50K-500K/项目
    - 20-50 个客户/年

  Tier 2（20-30%收入）: 标准化服务
    - 预配置的 AI workflow
    - $5K-20K/项目
    - 100-500 个客户/年

  Tier 3（<5%收入）: 专家社区（可选）
    - 开放给高认知用户
    - 免费或象征性收费
    - 建立口碑

收入目标: $5-10M ARR
团队: 5-10 个高认知管理者

关键: 用 Workbench 平台规模化服务能力
```

#### 阶段 3：知识网络（Year 3+）
```yaml
目标: 形成专家知识网络

模式:
  - 仍然以服务为主（不是 SaaS）
  - 但建立专家知识共享机制

知识交易:
  - 专家 A 贡献洞察 → 其他项目使用
  - 使用次数越多，专家收益越高
  - 平台抽成 30%，专家获得 70%

价值:
  - 知识复利效应
  - 专家网络效应
  - 降低服务成本

关键: 不是变成 SaaS，而是让服务更高效
```

#### 为什么服务模式更好？

**vs SaaS 订阅模式**：
```yaml
SaaS（已死）:
  收费: $500/月固定
  客户创造价值: $10M
  我们捕获: 0.06%
  问题: 客户要学工具，使用率低

服务模式:
  收费: $200K（按价值2%）
  客户创造价值: $10M
  我们捕获: 2%
  优势: 客户买结果，满意度高
```

**详见**：[ADR 004: 服务vs平台模式](../decisions/004-service-vs-platform-model.md)

### 3.2 定价策略

#### 方案 A：按效果付费（推荐）
```yaml
故障预测服务:
  基础费: $500/月
  效果提成: 故障成本节省的 20%

能效优化服务:
  基础费: $300/月
  效果提成: 能源成本节省的 25%

洞察发现服务:
  基础费: $1000/月
  效果提成:
    - 发现新服务机会 → 新收入的 10%
    - 发现重大问题 → 避免损失的 15%
```

**优势**：
- 利益绑定，客户愿意付费
- 定价与价值挂钩，可以收高价
- 客户认知："这是投资，不是成本"

**风险**：
- 需要量化效果（技术难度）
- 收入波动大（依赖客户业绩）

**适用场景**：中大型客户，有清晰 KPI

---

#### 方案 B：订阅 + 增值服务
```yaml
订阅费:
  基础版: $200/月（10 台设备）
  专业版: $800/月（50 台设备）
  企业版: $3000/月（无限设备 + 定制）

增值服务:
  专家咨询: $500/小时
  定制洞察: $5000/项目
  知识迁移: $10000/项目
```

**优势**：
- 收入稳定可预测
- 易于理解和销售

**风险**：
- 定价上限低（难以体现元层价值）
- 容易陷入"卖软件"思维

**适用场景**：中小型客户，追求确定性

---

#### 方案 C：混合模式（最佳实践）
```yaml
阶段 1（获客）:
  - 提供免费试用
  - 展示"AI 能帮你做什么"（可编码知识）
  - 定价：$99/月（入门版）

阶段 2（提升）:
  - 用户看到价值，愿意深度使用
  - 展示"AI 能帮你发现什么"（元层洞察）
  - 定价：$500/月 + 按效果分成

阶段 3（锁定）:
  - 用户依赖平台做战略决策
  - 平台成为"外部智囊团"
  - 定价：$2000/月 + 高比例分成 + 知识资产共享收益
```

Univers 服务方案：

## 四、知识积累战略

### 4.1 知识飞轮

```
专家使用平台
    ↓
产生洞察（元层知识）
    ↓
平台沉淀洞察
    ↓
其他专家受益
    ↓
更多专家使用平台
    ↓
（循环）
```

**关键设计**：
1. **低摩擦捕获**：专家在工作流中自然产生洞察，平台自动记录
2. **高价值回报**：洞察贡献者获得收益分成
3. **智能推荐**：AI 将相关洞察推送给需要的用户

### 4.2 知识资产化

#### 可编码知识
```python
# 示例：HVAC 参数优化规则
class OptimizationRule:
    """
    来源：100 个客户的历史数据
    验证：降低能耗平均 12%
    """
    def apply(self, scenario):
        if scenario.outdoor_temp < 5:
            return {
                "indoor_target": 20,  # 降低目标温度
                "humidity_target": 45,  # 降低湿度目标
                "reason": "低温场景下，降低目标可节能且不影响舒适度"
            }
```

**价值**：可以被 AI 执行，可以规模化

#### 情境知识
```python
# 示例：异常检测模式
class AnomalyPattern:
    """
    来源：张师傅 20 年经验
    场景：某品牌压缩机的早期故障特征
    """
    pattern = {
        "symptoms": [
            "启动电流波动 > 5%",
            "运行声音频率偏移 > 2Hz",
            "虽然温度控制正常"
        ],
        "diagnosis": "压缩机轴承磨损",
        "action": "建议 2 周内更换",
        "confidence": 0.85
    }
```

**价值**：可以被 AI 学习，但需要人类验证

---

#### 元层知识
```yaml
# 示例：行业洞察
insight:
  title: "HVAC 行业的真正痛点是故障预测，而非能效优化"

  来源:
    专家: 李工（15 年行业经验）
    数据: 200 个客户访谈
    时间: 2025-Q1

  论据:
    - 能耗成本占总成本 < 15%
    - 故障导致停机损失 > 能耗成本 3 倍
    - 客户愿意为"零故障"支付溢价 20%

  战略含义:
    - 重新定位产品：从"能效优化"到"故障预测"
    - 改变销售话术：强调"避免损失"而非"降低成本"
    - 调整研发重点：投入故障预测算法

  使用次数: 23
  价值创造: $450,000（23 个客户的合同增量）
  贡献者收益: $135,000（30% 分成）
```

**价值**：无法被 AI 生成，但可以被 AI 执行

**这是护城河所在**。

### 4.3 知识网络效应

#### 单点洞察
```
专家 A："某品牌空调在高湿环境下容易凝露"
价值：帮助 10 个客户避免问题
收益：$5,000
```

#### 网络洞察
```
专家 A："某品牌空调在高湿环境下容易凝露"
专家 B："我们发现预热 2 小时可以解决"
专家 C："其实根本原因是风道设计问题"

平台综合：
→ 形成"凝露问题完整解决方案"
→ 包括：诊断、临时缓解、根本解决

价值：帮助 100 个客户
收益：$50,000
分配：A/B/C 各得 30%，平台得 10%
```

**关键**：孤立的洞察价值有限，组合的洞察价值指数级增长

### 5.3 长期优势（5+ 年）- 真正的护城河

#### 元层洞察库
```
洞察 1："HVAC 行业痛点是故障预测"
  → 价值创造：$10M
  → 使用客户：500+
  → AI 能复制？❌ 不能

洞察 2："连锁酒店更在乎客户体验，而非能耗"
  → 价值创造：$5M
  → 使用客户：200+
  → AI 能复制？❌ 不能

洞察 3："工业客户愿意为零故障支付 20% 溢价"
  → 价值创造：$8M
  → 使用客户：300+
  → AI 能复制？❌ 不能
```

**为什么是护城河**：
1. **AI 无法生成**：元层洞察需要"质疑框架"，AI 做不到
2. **难以逆向**：看到结论，不等于理解推理过程
3. **复利效应**：洞察越多，平台价值越大

---

#### 缩放能力文化
- 组织了一批"善于缩放的人"（参见 [talent-strategy.md](talent-strategy.md)）
- 建立了"多尺度思维"的工作流程
- 形成了"质疑框架"的企业文化

**为什么是护城河**：
1. **难以模仿**：文化需要长期积累
2. **不可交易**：无法通过挖人/收购获得
3. **自我强化**：优秀的人吸引优秀的人

---

### 5.4 风险与应对

#### 风险 1：通用 AI 突破元层能力
```
风险：GPT-6 突然具备了"质疑框架"的能力
概率：低（5-10 年内）
影响：致命

应对：
- 持续跟踪 AI 前沿
- 如果 AI 真的突破，我们转型为"AI 训练师"
- 利用沉淀的元层知识数据，成为训练 AI 的数据源
```

---

#### 风险 2：竞争对手复制模式
```
风险：其他公司学习我们的"元层知识"模式
概率：高（2-3 年内）
影响：中等

应对：
- 先发优势：快速积累元层洞察库
- 网络效应：专家越多，价值越大
- 执行壁垒：有策略和有能力执行是两回事
```

---

#### 风险 3：客户不买单
```
风险：客户认为"洞察"不值那么多钱
概率：中（初期）
影响：高

应对：
- 阶段 1：先卖工具，建立信任
- 阶段 2：展示洞察价值（量化效果）
- 阶段 3：按效果付费，利益绑定
- 案例营销：用成功客户说服新客户
```

## 六、实施路线图

### Phase 1：验证（0-6 个月）

**目标**：证明"元层知识"有价值

**关键任务**：
1. 选择 3-5 个标杆客户
2. 深度访谈，发现元层洞察
3. 用 AI 执行洞察，量化效果
4. 形成案例

**成功标准**：
- 至少 1 个客户愿意为"洞察"付溢价
- 量化效果 > 3 倍投入

---

### Phase 2：规模化（6-18 个月）

**目标**：构建知识飞轮

**关键任务**：
1. 开发知识捕获系统（参见 [tech/knowledge-integration.md](../tech/knowledge-integration.md)）
2. 招募 20+ 领域专家
3. 沉淀 100+ 可复用洞察
4. 建立知识交易机制

**成功标准**：
- 50% 新客户使用"二手洞察"（非定制）
- 专家贡献者获得收益 > $10,000/年
- 知识复用率 > 5 次/洞察

---

### Phase 3：服务规模化（18-36 个月）

**目标**：规模化服务能力，而非成为 SaaS 平台

**关键任务**：
1. 规模化 AI 管理能力（同时服务 100+ 客户）
2. 建立专家社区（知识贡献和复用）
3. 标准化服务交付流程
4. 沉淀可复用元层洞察库

**成功标准**：
- 同时服务 100+ 客户（仍然是服务，不是 SaaS）
- 300+ 专家贡献者
- 1000+ 可复用洞察
- 服务边际成本降低 70%

**重要**：不是变成"客户使用的平台"，而是"我们用平台规模化服务"

## 八、总结

### 核心观点

1. **垂直知识价值重构**：可编码↓、情境↔、元层↑
2. **战略定位**：不卖工具，卖 AI 驱动的专业服务（元层洞察 + AI 执行）
3. **商业模式**：按价值收费的专业服务 → 服务规模化 → 知识网络
4. **护城河**：元层洞察库 + 缩放能力文化
5. **关键风险**：AI 突破、模仿、客户认知

### 下一步行动

1. **立即**：选择标杆客户，验证元层价值
2. **3 个月内**：量化至少 3 个元层洞察的效果
3. **6 个月内**：建立知识捕获系统
4. **12 个月内**：形成知识飞轮
