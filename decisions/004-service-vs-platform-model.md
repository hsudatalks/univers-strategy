# ADR 004: 服务模式 vs 平台模式 —— 谁来管理 AI？

## 状态
**草案** - 2025-11-04

## 背景

### 核心问题

在确立了"AI 执行者"的定位后（ADR 003），我们面临一个根本性的商业模式问题：

**在客户侧创造价值时，谁来管理 AI？**

```
选项 A：客户管理 AI（平台模式）
  Univers 提供平台
  → 客户购买平台
  → 客户自己管理 AI 执行者
  → 质量取决于客户的认知/品味/耐心

选项 B：我们管理 AI（服务模式）
  Univers 提供服务
  → 客户购买结果/服务
  → Univers 团队管理 AI 执行者
  → 质量由我们保证
```

### 为什么这个问题重要？

**来自 ADR 003 的洞察**：
```
质量 = (AI能力 × 迭代次数) × 管理者判断力

管理者判断力 = f(认知, 品味, 耐心)
```

**这意味着**：
- 高认知客户 + AI → 高质量产品
- 低认知客户 + AI → 低质量产品

**现实情况**：
- 大多数客户认知参差不齐
- 如果客户做不出好产品 → 会怪平台不好用
- 这会伤害 Univers 的品牌和口碑

### 新的可能性

**关键观察**：
1. **我们团队有高认知/品味/耐心**（自举验证已证明）
2. **AI 执行成本低**（比雇人便宜很多）
3. **我们管 AI 能保证质量**（经过验证的能力）

**那么为什么不是**：
```
Univers 团队用 AI 为客户创造价值？

优势：
  ✅ 质量有保证（我们的认知/品味/耐心）
  ✅ 成本可控（用 AI，不是雇大量人）
  ✅ 客户满意度高（买到的是结果，不是工具）
  ✅ 规模化可能（一个管理者可以管理 20 个 AI）
```

---

## 决策

**我们需要选择**：平台模式、服务模式，还是混合模式？

（这个 ADR 目前是**草案**，需要深入分析和讨论）

## 分析

### 模式对比

#### 模式 A：平台模式（客户管 AI）

**运作方式**：
```yaml
Univers 的角色: 平台提供商
客户的角色: AI 管理者

客户购买: Workbench 平台订阅
客户获得: 管理 AI 执行者的能力
客户负责: 自己管理 AI，验收结果，迭代优化

Univers 收入: 订阅费（按用户数或 AI 数量）
```

**优势**：
```
✅ 规模化潜力大
   - 一个平台可以服务无限客户
   - 不需要为每个客户配人力
   - 边际成本接近零

✅ 客户自主权高
   - 客户完全控制
   - 可以按自己的节奏和方向
   - 不依赖 Univers 团队的时间

✅ 技术壁垒清晰
   - 产品能力是核心
   - 可以持续迭代产品
   - 技术护城河
```

**劣势**：
```
❌ 质量不可控
   - 客户认知低 → 做不出好产品
   - 客户会怪平台"不好用"
   - 负面口碑伤害品牌

❌ 学习曲线陡峭
   - 客户需要学习如何管理 AI
   - 需要大量文档、教程、培训
   - 客户成功成本高

❌ 产品复杂度高
   - 需要适配各种场景
   - 需要强大的配置和自定义能力
   - 开发和维护成本高

❌ 价值感知弱
   - 客户付费买"工具"
   - 工具本身价值感知低
   - 定价天花板低
```

**适合的客户**：
```
✅ 高认知、高品味、有耐心的专家
✅ 有技术背景，理解 AI 局限
✅ 愿意投入时间学习和打磨

数量: 估计 < 10% 的潜在客户
```

**不适合的客户**：
```
❌ 低认知，看不出质量好坏
❌ 没耐心，迭代 3 次就放弃
❌ 不懂技术，不理解 AI 的工作方式

数量: 估计 > 50% 的潜在客户
```

---

#### 模式 B：服务模式（我们管 AI）

**运作方式**：
```yaml
Univers 的角色: 服务提供商
客户的角色: 需求方 / 验收方

客户购买: 具体的服务或项目
客户获得: 交付的结果（如：优化方案、自动化系统）
客户负责: 提供需求，验收结果

Univers 工作:
  - 理解客户需求
  - 用 AI 执行（Workbench 是内部工具）
  - 高速迭代打磨
  - 交付高质量结果

Univers 收入: 项目费或按效果付费
```

**优势**：
```
✅ 质量可控
   - 我们的认知/品味/耐心保证质量
   - 客户满意度高
   - 正面口碑积累

✅ 价值感知强
   - 客户买的是"结果"
   - 价值清晰，愿意付更高价格
   - 定价天花板高

✅ 客户门槛低
   - 客户不需要学习管理 AI
   - 不需要技术背景
   - 只需要提需求、验收结果

✅ 适用客户广
   - 不限于高认知客户
   - 所有有需求的客户都可以服务
   - 市场规模更大
```

**劣势**：
```
❌ 规模化困难
   - 每个客户需要人力投入
   - 虽然用 AI 降低执行成本，但管理者时间有限
   - 一个管理者最多管 20 个 AI（能同时服务多少客户？）

❌ 对团队要求高
   - 需要大量高认知/品味/耐心的管理者
   - 这样的人才稀缺且贵
   - 招聘和培训成本高

❌ 项目风险
   - 每个项目都有独特需求
   - 难以标准化
   - 可能遇到超出 AI 能力的需求

❌ 收入天花板
   - 受限于团队规模
   - 需要持续扩张团队才能增长
   - 不是纯软件的指数增长
```

**能服务的客户**：
```
✅ 所有有 HVAC 优化需求的客户
✅ 不限认知水平
✅ 愿意为结果付费

数量: 潜在市场很大
```

**盈利模型**：
```yaml
假设:
  一个高认知管理者: 管理 20 个 AI 执行者
  可以同时服务: 10-20 个客户项目（并发）

  平均项目价值: $10,000-50,000
  AI 成本: $500/月 × 20 = $10,000/月
  管理者成本: $15,000/月

  每月收入: 如果完成 5 个项目 × $20,000 = $100,000
  成本: $25,000/月
  毛利: $75,000/月

规模化:
  10 个管理者 → $750,000/月毛利
  但需要招到 10 个高认知管理者（难！）
```

---

#### 模式 C：混合模式

**分层服务**：

**层级 1：顶级服务（我们管 AI）**
```yaml
目标客户: 大客户、复杂项目、高价值需求
服务方式: Univers 团队用 AI 为客户交付
定价: 高（$50,000-500,000/项目）
客户数量: 少（每年 10-50 个）
团队投入: 高（需要高认知管理者）

价值:
  - 质量最高
  - 客户满意度最高
  - 单客户收入高
```

**层级 2：自助平台（客户管 AI）**
```yaml
目标客户: 高认知的专家、技术型创业者
服务方式: 客户自己使用 Workbench 平台
定价: 中（$500-5,000/月订阅）
客户数量: 多（理论上无限）
团队投入: 低（只需维护平台）

价值:
  - 规模化潜力大
  - 边际成本低
  - 培养高认知用户社区
```

**层级 3：标准化产品（AI 自动化）**
```yaml
目标客户: 小客户、标准化需求
服务方式: 预配置的自动化方案
定价: 低（$100-500/月）
客户数量: 非常多
团队投入: 极低（完全自动化）

价值:
  - 覆盖长尾市场
  - 几乎零边际成本
  - 品牌认知
```

**混合模式的优势**：
```
✅ 覆盖不同客户群
✅ 收入来源多样化
✅ 高端服务建立品牌和口碑
✅ 平台模式实现规模化
✅ 标准化产品覆盖长尾
```

**混合模式的劣势**：
```
❌ 复杂度极高（三种商业模式）
❌ 资源分散
❌ 定位混乱（我们到底是什么？）
❌ 组织挑战（需要不同类型的团队）
```

### 战略考量

#### 考量 1：我们的核心能力是什么？

**如果核心能力是"用 AI 创造价值"**：
```
→ 应该选服务模式
→ 直接为客户创造价值
→ Workbench 是内部工具
→ 类比：咨询公司用内部工具服务客户
```

**如果核心能力是"构建让别人能用 AI 创造价值的平台"**：
```
→ 应该选平台模式
→ 赋能客户自己创造价值
→ Workbench 是产品
→ 类比：Salesforce、AWS
```

**目前情况**：
- 我们自己用 AI 开发了 Workbench（服务于自己）
- 我们还没有客户（尚未验证）
- 两种能力都可能具备

---

#### 考量 2：规模化路径

**服务模式的规模化**：
```
限制因素: 高认知管理者的数量
规模化路径:
  1. 招聘更多高认知管理者（难，慢）
  2. 培训普通管理者 → 高认知（可能，但需时间）
  3. 标准化部分服务 → 降低认知要求（折中）
  4. 最终过渡到平台？（长期）

增长曲线: 线性到亚线性（受人力限制）
```

**平台模式的规模化**：
```
限制因素: 产品能力、市场接受度
规模化路径:
  1. 打磨产品，降低学习曲线
  2. 建立客户成功体系
  3. 社区驱动（用户互助）
  4. 网络效应（优秀用户分享经验）

增长曲线: 指数增长（软件特性）
```

**混合模式的规模化**：
```
短期: 用服务建立口碑和品牌
中期: 将服务中的最佳实践沉淀到平台
长期: 平台成为主要收入来源

增长曲线: 先慢后快
```

---

#### 考量 3：市场定位与竞争

**服务模式的竞争**：
```
竞争对手: 传统咨询公司、工程服务商
差异化: 我们用 AI（成本低、速度快）

风险:
  - 客户可能认为"就是外包"
  - 难以建立技术壁垒
  - 容易被模仿
```

**平台模式的竞争**：
```
竞争对手: 其他 AI 平台、低代码平台
差异化: 专注垂直领域、人机协作设计

优势:
  - 技术壁垒更高
  - 网络效应
  - 持续迭代产品
```

---

#### 考量 4：与核心哲学的对齐

回顾 PHILOSOPHY.md：
```
"AI 的本质：扩展而非替代"
"赋能而非代替"
"让用户变得更强大，而不是更依赖"
```

**服务模式**：
```
⚠️ 可能与哲学冲突
   - 客户依赖我们交付
   - 客户没有变得"更强大"
   - 更像是"替代"而非"扩展"

✅ 但也可以对齐
   - 如果服务过程中培养客户能力
   - 如果最终目标是让客户能自己做
   - 类似"顾问 + 交付"模式
```

**平台模式**：
```
✅ 完全对齐
   - 赋能客户
   - 客户掌握工具后变强大
   - 符合"扩展"的理念

⚠️ 但有现实挑战
   - 不是所有客户都能被赋能
   - 低认知客户用不好平台
```

### 实际案例分析

让我们看看其他公司的选择：

#### 案例 1：OpenAI

**模式选择**：平台模式（API）
```
提供: GPT API
客户: 开发者自己调用 API 构建应用
结果:
  ✅ 指数增长（ChatGPT 1亿用户）
  ✅ 生态繁荣
  ❌ 但很多应用质量差（prompt 写得差）
```

**如果 OpenAI 选服务模式**：
```
可能: OpenAI 团队为企业客户定制 AI 方案
优势: 质量可控
劣势: 规模化困难，不会有今天的影响力
```

---

#### 案例 2：McKinsey + AI

**模式选择**：服务模式
```
提供: 咨询服务（内部用 AI 工具）
客户: 企业购买咨询项目
结果:
  ✅ 高价值（项目费几百万美元）
  ✅ 质量保证
  ❌ 规模化受限于顾问数量
  ❌ 增长慢
```

---

#### 案例 3：Salesforce

**模式选择**：平台模式
```
提供: CRM 平台
客户: 企业自己配置和使用
关键:
  - 提供培训和认证
  - Trailhead 学习平台
  - 强大的社区
  - AppExchange 生态

结果:
  ✅ 成为行业标准
  ✅ 网络效应
  ✅ $300B 市值
```

**启示**：
- 平台模式可以成功
- 但需要强大的客户成功体系
- 需要降低学习曲线

---

#### 案例 4：Palantir

**模式选择**：混合模式
```
初期（2003-2015）：纯服务
  - 团队驻场帮客户用 Palantir
  - 高度定制化
  - 客户满意，但规模化慢

转型（2015-2020）：平台化
  - 产品 Foundry 更易用
  - 客户可以自己配置
  - 但仍提供高端服务

现在：混合
  - 大客户：服务模式（驻场团队）
  - 中小客户：平台模式（自助）

结果: 成功上市，$40B 市值
```

**启示**：
- 可以从服务开始，逐步平台化
- 服务建立口碑和案例
- 将服务中的模式沉淀到平台

### 关键权衡

| 维度 | 服务模式 | 平台模式 | 混合模式 |
|-----|---------|---------|---------|
| **质量控制** | ✅✅✅ 完全可控 | ❌ 依赖客户 | ✅✅ 高端可控 |
| **规模化** | ❌ 线性增长 | ✅✅✅ 指数增长 | ✅ 逐步提升 |
| **客户门槛** | ✅✅ 低，所有人 | ❌ 高，需高认知 | ✅✅ 分层覆盖 |
| **价值感知** | ✅✅✅ 强，买结果 | ✅ 中，买工具 | ✅✅ 分层定价 |
| **技术壁垒** | ❌ 弱，易模仿 | ✅✅ 强，产品护城河 | ✅ 平台有壁垒 |
| **收入天花板** | ❌ 受人力限制 | ✅✅✅ 理论无限 | ✅✅ 高 |
| **组织复杂度** | ✅ 简单 | ✅ 简单 | ❌ 复杂 |
| **与哲学对齐** | ⚠️ 有争议 | ✅✅ 完全对齐 | ✅ 可以对齐 |
| **初期验证** | ✅✅ 快速验证 | ❌ 需要时间 | ✅✅ 快速 + 长期 |
| **团队要求** | 高认知管理者 | 产品工程师 | 两种都需要 |

## 推荐方案（待讨论）

基于以上分析，我倾向于**分阶段混合模式**：

### 阶段 1：服务优先（Year 1）

**做什么**：
```yaml
主要: 用 AI 为客户提供 HVAC 优化服务
目标客户: 10-20 个愿意付费的早期客户
团队: 创始人 + 2-3 个高认知管理者
工具: Workbench（内部工具）

关键目标:
  ✅ 验证价值（客户愿意为结果付费）
  ✅ 建立口碑和案例
  ✅ 打磨内部工具（Workbench）
  ✅ 积累垂直领域知识
  ✅ 探索可标准化的模式

收入目标: $500K-1M ARR
```

**为什么这样做**：
- 快速验证市场需求
- 质量可控，建立品牌
- 不需要复杂的产品
- 专注于创造价值

---

### 阶段 2：服务 + 平台（Year 2-3）

**做什么**：
```yaml
主要:
  1. 继续提供高端服务（大客户）
  2. 开放 Workbench 给高认知客户（自助平台）

分层:
  层级 1: 高端服务（我们管 AI）
    - 大客户、复杂项目
    - $50K-500K/项目
    - 20-50 个客户/年

  层级 2: 平台订阅（客户管 AI）
    - 高认知专家、技术型创业者
    - $500-5K/月
    - 100-500 个客户

团队:
  - 服务团队: 5-10 个高认知管理者
  - 平台团队: 10-15 个工程师

关键目标:
  ✅ 将服务中的模式沉淀到平台
  ✅ 建立客户成功体系（培训、文档、社区）
  ✅ 验证平台的自助可行性
  ✅ 探索网络效应

收入目标: $5-10M ARR
  - 服务: $3-5M
  - 平台: $2-5M
```

---

### 阶段 3：平台为主（Year 4+）

**做什么**：
```yaml
主要: Workbench 成为主要产品
服务: 保留高端服务（品牌和口碑）

分层:
  层级 1: 顶级服务（10-20%收入）
  层级 2: 平台订阅（60-70%收入）
  层级 3: 标准化自动化产品（10-20%收入）

团队:
  - 平台团队为主（80%）
  - 服务团队保留精英（20%）

关键目标:
  ✅ 规模化增长
  ✅ 网络效应
  ✅ 生态繁荣

收入目标: $50M+ ARR
```

---

### 为什么这个路径可行？

**1. 符合 Palantir 的成功路径**
```
Palantir:
  初期服务 → 积累经验 → 产品化 → 平台 → 上市

我们:
  初期服务 → 验证价值 → 沉淀平台 → 规模化
```

**2. 兼顾短期和长期**
```
短期: 服务快速验证和盈利
长期: 平台指数增长
```

**3. 降低风险**
```
风险 1: 市场不接受平台
  → 服务已经验证了价值

风险 2: 平台学习曲线太陡
  → 服务中积累的经验可以沉淀到平台

风险 3: 服务规模化困难
  → 逐步过渡到平台
```

**4. 符合核心哲学**
```
阶段 1-2:
  "我们用 AI 扩展能力"（体现价值）

阶段 2-3:
  "赋能客户用 AI 扩展能力"（实现使命）
```

## 重要趋势："SaaS 已死"论

### 背景

2023-2024 年，"SaaS is Dead"（SaaS 已死）成为硅谷创投圈热门论调。这对我们的战略有重要影响。

### 核心论据

**1. 从"工具"到"结果"**
```
传统 SaaS: 卖工具，客户自己用
AI 时代: 卖结果，AI 帮客户完成任务

例子:
  - 不需要 CRM，AI 直接帮管理客户
  - 不需要 Excel，AI 直接给分析结果
```

**2. 客户不想学习工具**
```yaml
SaaS 痛点:
  - 培训成本高
  - 平均功能使用率: 30-40%
  - 流失原因: "太复杂" 40%，"没时间学" 30%

客户真正想要:
  - 解决问题，不是学工具
  - 买结果，不是买软件
```

**3. 价值捕获天花板**
```yaml
SaaS 订阅模式:
  - 客户创造 $10M 价值
  - 收费: $300/月固定
  - 价值捕获: 0.036%

AI 服务模式:
  - 客户创造 $10M 价值
  - 收费: $200K-500K（按价值）
  - 价值捕获: 2-5%
  - 执行成本远低于传统咨询
```

**4. AI 降低了 SaaS 壁垒**
```
传统: 功能开发 18 个月 → 护城河
AI 时代: AI 快速开发 3 个月 → 壁垒降低
```

**5. OpenAI 平台化核心能力**
```yaml
问题:
  SaaS 只是包装 OpenAI
  → OpenAI 拿走大部分成本
  → SaaS 利润被压缩（毛利 25% vs 传统 90%）
  → OpenAI 随时可能推出竞品
```

**6. 从"软件"到"服务"的回归**
```yaml
历史轮回:
  1990s-2000s: 传统服务（不可规模化）
  2000s-2020s: SaaS（可规模化，但价值捕获受限）
  2020s-?: AI 驱动的服务（可规模化 + 高价值捕获）

关键: AI 让"服务"可以规模化
```

### 对 Univers 的影响

**✅ 好消息：我们的方向是对的**

原计划（服务优先）完全契合趋势：
```yaml
阶段 1: 服务模式
  - 客户买结果，不是工具
  - 按价值/项目收费
  - 我们管 AI，客户不需要学习

→ 这正是"SaaS 已死"论所倡导的方向
```

**⚠️ 需要修正：重新思考"最终过渡到平台"**

原计划假设：
```
Year 1-2: 服务为主
Year 3+: 逐步过渡到平台为主
Year 4+: 平台成为主要收入来源
```

问题：
- 如果 SaaS 真的已死，"平台为主"还对吗？
- 客户可能永远不想学工具
- 应该永久保持"服务为主"？

**修正后的战略方向：**

**方案：重新定义"平台"**

不是"让客户用的 SaaS 平台"，而是"我们用来规模化服务的内部平台"

```yaml
客户视角:
  - 永远是"买服务/结果"
  - 不需要知道我们怎么做的
  - 不需要学习工具

我们内部:
  - Workbench 是内部运营平台
  - 让一个管理者管理 20 个 AI
  - 从而同时服务 10-20 个客户
  - 实现服务的"伪规模化"

类比:
  - Uber: 客户看到的是"打车服务"
  - DoorDash: 客户看到的是"送餐服务"
  - Univers: 客户看到的是"HVAC 优化服务"

关键: 平台不是产品，而是基础设施
```

**修订后的分层策略：**

```yaml
Tier 1: 高端服务（我们管 AI）
  - 大客户、复杂项目
  - $50K-500K/项目，按价值收费
  - 收入占比: 60-70%

Tier 2: 标准化服务（AI 自动化）
  - 标准需求，预配置 workflow
  - $5K-20K/项目
  - 收入占比: 20-30%

Tier 3: 专家社区（可选，次要）
  - 开放给极少数高认知用户
  - 免费或象征性收费
  - 目的：建立社区和口碑
  - 收入占比: < 5%

关键变化:
  - 不再以"过渡到平台"为目标
  - 永远以服务为主
  - Workbench 主要是内部工具
  - 只对极少数专家开放（建立社区）
```

**定价策略调整：**

```yaml
主要收入: 按价值/结果收费
  - 能源节省分成: 节省 $100K → 收费 $20K（20%）
  - 项目固定费: $10K-1M（按规模）
  - ROI 明确，客户愿意付

次要收入: 平台订阅（如果提供）
  - 仅针对高认知专家
  - 可能免费（建立社区）
  - 不期望主要收入来源
```

**Workbench 定位调整：**

```yaml
主要定位: 内部运营平台（90%）
  - Univers 团队用来服务客户
  - 提高效率和质量
  - 不对外销售

次要定位: 专家社区工具（10%）
  - 开放给少数高端用户
  - 建立专家网络
  - 口碑和反馈来源
```

### 案例参考

**Harvey AI（法律）**：
```
模式: AI 驱动的法律服务（不是 SaaS）
收费: 按案件复杂度 $5K-50K
估值: $1.5B（2024）
```

**Hebbia（企业研究）**：
```
模式: AI 驱动的研究服务（不是 SaaS）
收费: 按项目 $10K-100K
估值: $700M（2024）
```

### 核心结论

**Univers 应该成为**：
- ✅ AI 驱动的 HVAC 优化服务公司
- ✅ 用 Workbench 内部平台规模化服务
- ✅ 按价值/结果收费
- ❌ 不是 SaaS 平台公司

## 开放问题

1. **服务的标准化程度**：我们能把多少比例的服务标准化？
2. **规模化路径**：如何在不依赖纯 SaaS 的情况下实现规模化？
3. **人才供给**：能招到足够多的高认知管理者吗？如何培养？
4. **定价策略**：如何向客户传达"按价值收费"的模式？
5. **Workbench 的边界**：哪些功能应该对外开放？哪些保持内部？
