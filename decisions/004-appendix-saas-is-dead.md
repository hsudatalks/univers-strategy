# 附录：SaaS 已死论与 Univers 战略

## 背景

2023-2024 年，"SaaS is Dead"（SaaS 已死）成为硅谷和创投圈的热门论调。这个趋势对我们在 ADR 004 中制定的商业模式有重大影响。

---

## "SaaS 已死"的核心论据

### 论据 1：从"工具"到"结果"的转变

**传统 SaaS 模式**：
```yaml
产品: 提供工具（软件）
客户购买: 订阅使用权
客户需要:
  - 学习如何使用工具
  - 自己操作工具完成任务
  - 雇人使用这个工具

例子:
  - Salesforce: 客户买CRM工具，自己管理销售
  - Photoshop: 客户买设计工具，自己做设计
  - Excel: 客户买表格工具，自己做分析
```

**AI 时代的转变**：
```yaml
产品: 提供结果（服务）
客户购买: 完成的任务
客户不需要:
  - 学习工具
  - 雇人操作
  - 管理流程

例子:
  - 不需要CRM，AI直接帮你管理客户
  - 不需要Photoshop，AI直接生成设计
  - 不需要Excel，AI直接给你分析结果
```

**关键变化**：
```
传统: 我卖你工具，你自己干活
现在: 我帮你干活，你买结果
```

---

### 论据 2：客户不想学习工具

**SaaS 的痛点**：
```
购买 SaaS → 培训团队 → 改变工作流 → 长期学习曲线

问题:
  - 培训成本高
  - 采用率低（很多功能不用）
  - 需要持续维护和更新知识
  - 员工离职后知识流失
```

**客户真正想要的**：
```
我有一个问题 → 你帮我解决 → 我付费

不关心:
  - 你用什么工具
  - 怎么做的
  - 需要多少步骤
```

**数据支持**：
```yaml
平均 SaaS 采用率:
  - 购买的功能使用率: 30-40%
  - 团队真正熟练使用: < 20%
  - 12个月后仍在活跃使用: 60%

客户流失原因:
  1. "太复杂，学不会" - 40%
  2. "没时间学" - 30%
  3. "团队不用" - 20%
```

---

### 论据 3：价值捕获的天花板

**SaaS 订阅模式的限制**：
```yaml
传统定价:
  - 按用户数: $50/用户/月
  - 按功能层级: 基础版$99/月，专业版$299/月
  - 按使用量: $0.01/API调用

问题:
  ✅ 可预测的收入
  ❌ 价值捕获有天花板
  ❌ 客户创造巨大价值，但你只收固定订阅费
```

**AI 服务的价值捕获**：
```yaml
按结果/价值定价:
  - 帮客户优化HVAC，节省$100,000/年 → 收费$20,000
  - 帮客户生成设计，价值$50,000 → 收费$10,000
  - 帮客户找到新客户，带来$500,000收入 → 收费$50,000

优势:
  ✅ 捕获创造价值的一部分
  ✅ 价值越大，收费越多
  ✅ 与客户利益对齐
```

**例子对比**：

**SaaS 模式（Salesforce）**：
```
客户用 Salesforce 签下 $10M 大单
→ Salesforce 收费: $300/月（固定）
→ 价值捕获: 0.036%
```

**服务模式（传统咨询）**：
```
McKinsey 帮客户做战略，带来 $10M 收入
→ McKinsey 收费: $500K-1M（按项目）
→ 价值捕获: 5-10%
```

**AI 服务模式（可能）**：
```
AI 帮客户完成任务，创造 $10M 价值
→ 收费: $200K-500K（按价值）
→ 价值捕获: 2-5%
→ 但执行成本远低于 McKinsey
```

### 论据 4：AI 降低了 SaaS 的壁垒

**传统 SaaS 的护城河**：
```yaml
壁垒:
  - 复杂的功能开发（需要多年）
  - 积累的用户数据
  - 网络效应
  - 客户切换成本（学习曲线）
```

**AI 时代的变化**：
```yaml
开发壁垒降低:
  - AI 可以快速开发功能
  - 从18个月缩短到3个月
  - 创业公司可以快速追赶

数据壁垒弱化:
  - 大语言模型已经有通用能力
  - 不需要从零训练
  - 少量数据就能 fine-tune

切换成本降低:
  - 如果不需要学习工具
  - 切换服务提供商更容易
  - 客户忠诚度下降
```

**案例**：
```
传统:
  Salesforce 花 20 年建立护城河
  → 新竞争对手需要 5-10 年追赶

AI 时代:
  新的 AI CRM 用 GPT-4 + 6 个月开发
  → 功能可能已经接近 Salesforce
  → Salesforce 的优势被压缩
```

---

### 论据 5：OpenAI 等平台化核心能力

**问题**：
```yaml
SaaS 的困境:
  核心能力（AI）来自 OpenAI
  → SaaS 只是包装层（UI + prompt）
  → OpenAI 拿走大部分成本
  → SaaS 利润被压缩

例子:
  AI 写作助手:
    收费: $20/月
    OpenAI 成本: $15/月
    → 毛利润: $5/月（25%）

  vs 传统 SaaS:
    收费: $100/月
    成本: $10/月
    → 毛利润: $90/月（90%）
```

**更严重的问题**：
```
OpenAI 随时可以推出类似功能
→ 直接竞争
→ SaaS 的差异化消失
```

**案例：Jasper AI**：
```
2022年:
  - AI 写作工具
  - 估值 $1.5B
  - 收入 $100M ARR

2023年:
  - ChatGPT 推出，更便宜更好
  - Jasper 估值暴跌
  - 被迫大幅降价

教训: 只是包装 OpenAI 的 SaaS 很危险
```

### 论据 6：从"软件"到"服务"的回归

**历史轮回**：
```
1990s-2000s:
  - 传统服务（咨询、外包）
  - 收费高，但不可规模化

2000s-2020s:
  - SaaS 崛起
  - "软件正在吞噬世界"
  - 可规模化，但价值捕获受限

2020s-?:
  - AI 时代
  - "服务重新崛起，但AI驱动"
  - 可规模化（AI执行） + 高价值捕获（按结果收费）
```

**新范式**：
```yaml
传统服务:
  可规模化: ❌（受人力限制）
  价值捕获: ✅✅（高，按价值收费）

SaaS:
  可规模化: ✅✅（软件边际成本为零）
  价值捕获: ⚠️（低，订阅天花板）

AI 驱动的服务:
  可规模化: ✅（AI执行，人管理）
  价值捕获: ✅✅（按结果/价值收费）
```

**关键洞察**：
```
SaaS 试图用"软件"替代"服务"
但客户真正想要的是"结果"，不是"工具"

AI 让"服务"可以规模化
所以不再需要妥协为"软件"
```

### Marc Andreessen (a16z)

```yaml

  AI 服务公司: 10-15x ARR（如果毛利高）
```

## 对 Univers 的启示

### 启示 1：我们的 ADR 004 方向正确

**我们的选择（ADR 004）**：
```
阶段 1: 服务优先（我们管 AI，客户买结果）
阶段 2: 服务 + 平台
阶段 3: 平台为主？
```

**与"SaaS已死"论调对比**：
```
✅ 阶段 1 完全对齐
   - 服务模式 = 卖结果
   - 按项目/价值收费
   - 客户不需要学习工具

⚠️ 阶段 3 需要重新思考
   - 如果 SaaS 真的已死
   - "最终过渡到平台"还对吗？
   - 也许应该永久保持服务为主？
```

---

### 启示 2：重新定义"平台"

**传统 SaaS 平台**（可能已死的模式）：
```yaml
客户购买: 软件订阅
客户做: 自己操作软件
我们提供: 工具
收费方式: $X/月固定
价值捕获: 低
```

**新型"AI 服务平台"**（可能的未来）：
```yaml
客户购买: 完成的任务/结果
客户做: 提需求，验收结果
我们提供: 结果（AI执行，我们管理）
收费方式: 按价值/结果
价值捕获: 高

关键:
  - "平台"不是给客户用的工具
  - 而是我们内部用来规模化服务的基础设施
```

**例子：对比**

**Salesforce（传统SaaS平台）**：
```
客户买: Salesforce 软件使用权
客户学: 如何用 Salesforce
客户做: 自己管理销售流程
Salesforce 收: $300/用户/月
```

**AI 销售服务（新模式）**：
```
客户买: "帮我找到并跟进潜在客户"
客户学: 不需要学任何软件
我们做: 用 AI + 内部平台完成任务
我们收: 按成交客户数，$500/成交
```

**区别**：
- Salesforce 是"SaaS 平台"（客户用）
- 我们是"AI 服务平台"（我们用来规模化服务）

### 启示 3：ADR 004 的修正方向

**原计划（ADR 004）**：
```
Year 1: 服务（我们管 AI）
Year 2-3: 服务 + 平台（部分客户自己管 AI）
Year 4+: 平台为主（大部分客户自己管 AI）
```

**修正后（考虑 SaaS 已死）**：

**方案 A：永久服务为主**
```yaml
Year 1-2: 纯服务
  - 我们用 AI 为客户创造价值
  - 按结果/价值收费
  - 客户不需要学习任何工具

Year 3+: 服务 + 自助（分层）
  高端客户（80%收入）:
    - 服务模式（我们管 AI）
    - 按价值收费

  长尾客户（20%收入）:
    - 自助平台（客户管 AI）
    - 但只针对高认知用户
    - 订阅费较低

核心: 永远以服务为主
```

**方案 B：重新定义平台**
```yaml
不是"让客户用我们的平台"
而是"我们用平台规模化服务"

客户视角:
  - 永远是"买服务/结果"
  - 不需要知道我们怎么做的
  - 不需要学习工具

我们内部:
  - Workbench 是我们的内部平台
  - 让一个管理者可以管理 20 个 AI
  - 从而同时服务 10-20 个客户
  - 实现服务的"伪规模化"

类比:
  - Uber: 客户看到的是"打车服务"，不是"平台"
  - DoorDash: 客户看到的是"送餐服务"，不是"平台"
  - Univers: 客户看到的是"HVAC优化服务"，不是"平台"
```

**方案 C：混合定位**
```yaml
分层策略:

Tier 1: 高端服务（我们管 AI）
  - 大客户、复杂项目
  - $50K-500K/项目
  - 按价值收费
  - 20-50 客户/年
  - 收入: 60-70%

Tier 2: 标准服务（AI 自动化）
  - 标准化需求
  - $5K-20K/项目
  - 预配置的 AI workflow
  - 100-500 客户/年
  - 收入: 20-30%

Tier 3: 自助平台（客户管 AI）
  - 高认知专家
  - $500-2K/月订阅
  - 只针对特定用户群
  - 收入: 5-10%

核心变化:
  - Tier 3（纯 SaaS）占比很小
  - 主要收入来自服务（Tier 1+2）
```

### 启示 4：Workbench 的定位变化

**原定位（ADR 004）**：
```
Workbench = 产品
→ 最终要卖给客户
→ 客户自己用来管理 AI
→ 按订阅收费
```

**新定位（考虑 SaaS 已死）**：

**选项 A：纯内部工具**
```yaml
Workbench = 内部运营平台

用途:
  - 让 Univers 团队高效管理 AI 执行者
  - 同时服务多个客户项目
  - 标准化服务流程

不对外:
  - 客户不需要看到 Workbench
  - 客户只需要提需求 + 验收结果

类比:
  - Uber 的司机调度系统（内部用）
  - DoorDash 的物流系统（内部用）
  - 麦肯锡的内部知识库（内部用）

价值:
  - 让服务可以规模化
  - 降低每个客户的服务成本
  - 提高质量一致性
```

**选项 B：开放给极少数用户**
```yaml
Workbench = 主要是内部工具 + 小部分外部

内部（主要用途 90%）:
  - Univers 团队用来服务客户

外部（次要用途 10%）:
  - 只对高认知、高品味、有耐心的用户开放
  - 作为"专家社区"
  - 可能不收费或象征性收费
  - 目的是建立社区和口碑

价值:
  - 主要价值仍是规模化服务
  - 次要价值是建立专家网络
```

**推荐：选项 B**

原因：
1. 主要定位是内部工具（符合"SaaS已死"）
2. 但保留开放给少数高端用户（建立社区）
3. 不期望从 SaaS 订阅中获得主要收入
4. 灵活性高，未来可调整

### 启示 5：定价策略的根本变化

**原定价（ADR 004，SaaS思维）**：
```yaml
平台订阅:
  - 基础版: $500/月（1个 AI 执行者）
  - 专业版: $2,000/月（5个 AI 执行者）
  - 企业版: $6,000/月（20个 AI 执行者）

服务项目:
  - $50K-500K/项目
```

**新定价（服务优先思维）**：

**主要收入：服务（按价值）**
```yaml
HVAC 优化服务:

  价值分成模式:
    客户节省 $100K/年能源成本
    → 我们收费 $20K（20% 分成）

    客户提升 $500K/年租金收入（更舒适的建筑）
    → 我们收费 $50K（10% 分成）

  固定项目模式:
    小型项目（1-2栋建筑）: $10K-30K
    中型项目（5-10栋建筑）: $50K-150K
    大型项目（20+栋建筑）: $200K-1M

定价逻辑:
  ✅ 基于创造的价值
  ✅ 客户愿意付（有明确 ROI）
  ✅ 捕获价值的一部分
  ❌ 不是固定订阅费
```

**次要收入：平台（如果提供）**
```yaml
仅针对特定用户:
  - 高认知专家
  - 想要学习和自主控制的人

定价:
  - 可能免费（建立社区）
  - 或象征性收费 $100-500/月
  - 不期望这是主要收入来源
```

### 启示 6：竞争定位的变化

**原定位（ADR 004）**：
```
我们 vs 其他 AI 平台
→ 功能对比
→ 易用性对比
→ 价格对比
```

**新定位（服务模式）**：
```
我们 vs 传统服务商

对比对象:
  - 传统 HVAC 咨询公司
  - 工程服务公司
  - 系统集成商

差异化:
  ✅ 我们用 AI（成本低、速度快）
  ✅ 但客户感知的是"更好的服务"
  ✅ 不是"更便宜的软件"

案例:
  传统咨询: $500K，6 个月，10 个顾问
  我们: $200K，3 个月，2 个管理者 + 20 个 AI

  客户看到: 更快、更便宜、质量相当
  我们的利润率: 更高（AI 成本低）
```

## 最终建议：修订 ADR 004

### 核心修改

**从**：
```
阶段 1: 服务
阶段 2: 服务 + 平台
阶段 3: 平台为主
```

**改为**：
```
阶段 1-2: 服务为主（AI 驱动的专业服务）
  - 我们管 AI，客户买结果
  - 按价值/项目收费
  - Workbench 是内部工具

阶段 3+: 规模化服务（仍然是服务，不是 SaaS）
  - 通过 AI + 平台让服务规模化
  - 一个管理者服务更多客户
  - 客户仍然买结果，不是工具
  - 可能开放给极少数高端用户（社区）

永远不会:
  ❌ 变成纯 SaaS 订阅模式
  ❌ 主要收入来自平台订阅
  ❌ 让大部分客户自己管 AI
```

### 关键原则

1. **客户买的是结果，不是工具**
   - 服务模式是核心
   - SaaS 订阅（如果有）是次要的

2. **Workbench 主要是内部工具**
   - 让我们能规模化服务
   - 不是主要的对外产品

3. **按价值收费，不是按订阅**
   - 捕获创造价值的一部分
   - 不受订阅天花板限制

4. **竞争对手是传统服务商，不是 SaaS**
   - 我们的优势是 AI（成本低、速度快）
   - 但定位是"更好的服务"

5. **"平台"的意义是规模化服务，不是卖软件**
   - 类似 Uber（服务公司）而非 Salesforce（软件公司）

## 结论

**"SaaS 已死"对 Univers 的影响**：

✅ **好消息**：
1. 我们的 ADR 004 方向是对的（服务优先）
2. 服务模式价值捕获更高
3. 符合市场趋势

⚠️ **需要修正**：
1. 不应该以"最终过渡到 SaaS 平台"为目标
2. Workbench 应该主要是内部工具，而非对外产品
3. 定价应该基于价值，而非订阅
4. 竞争定位是"更好的服务"，而非"更好的软件"

**核心洞察**：
```
SaaS 的价值主张: "我卖你工具，你自己干活"
→ 这个价值主张在 AI 时代不再成立

AI 服务的价值主张: "我用 AI 帮你干活，你买结果"
→ 这才是未来
```

**Univers 应该成为**：
- ✅ AI 驱动的 HVAC 优化服务公司
- ✅ 用 Workbench 内部平台规模化服务
- ✅ 按价值/结果收费
- ❌ 不是 SaaS 平台公司
