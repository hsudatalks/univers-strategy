# ADR 002: 垂直知识管理战略

## 状态
**已采纳** - 2025-11-02

## 背景

### 问题陈述

在 AI 时代，**垂直领域知识的价值正在重构**。我们面临以下关键问题：

1. **AI 能力边界不清晰**
   - AI 在垂直领域能做什么、不能做什么？
   - 哪些知识应该交给 AI，哪些应该保留给人类？
   - 如何定义人类专家在 AI 时代的价值？

2. **商业模式困境**
   - 如果 AI 能学会领域知识，我们还能卖什么？
   - 如何避免沦为"可替代的工具供应商"？
   - 护城河在哪里？

3. **知识管理挑战**
   - 如何让领域专家愿意分享知识？（而不是担心被替代）
   - 如何在工作流中低摩擦地捕获知识？
   - 如何让知识产生复利效应？

### 相关背景

**公司定位**：Univers 是一家 AIoT 物联网平台公司
- Workbench：AI + 人类的工作台
- Skills：连接 AI 和 Workbench 的基础能力
- Operation：AI 工作的场所（如 HVAC 项目）

**核心哲学**（参见 [PHILOSOPHY.md](../PHILOSOPHY.md)）：
- 人类的独特价值在于**上下文自由缩放**能力
- AI 的局限在于**被动缩放**，无法主动切换尺度和质疑框架
- 因此，我们的战略应该是：让人类做"元层决策"（what），让 AI 做"执行"（how）

**时间点**：2025 年 11 月
- GPT-4、Claude 等大模型已经广泛应用
- AI 在可编码知识上已经超越人类
- 但在元层思考上仍然缺失

### 决策驱动因素

**战略层面**：
- 需要明确"我们的护城河是什么"
- 需要定义"人类专家的价值主张"
- 需要找到"AI 无法替代的竞争优势"

**业务层面**：
- 需要设计可持续的商业模式（不只是卖软件）
- 需要构建网络效应（知识平台 > 单点工具）
- 需要激励专家贡献知识

**技术层面**：
- 需要决定技术栈（如何管理不同类型的知识）
- 需要设计系统架构（支持知识捕获、执行、演化）
- 需要弥补 AI 的缩放局限

---

## 决策

我们决定采用**三层知识管理战略**：

### 1. 知识分层模型

将垂直领域知识分为三个层次，对应不同的 AI 能力和人类价值：

#### 第一层：可编码知识（Codifiable Knowledge）
```yaml
定义: 可以明确表达的规则、公式、流程、标准

AI 能力:
  学习: ✅ 容易
  执行: ✅ 比人更好
  泛化: ✅ 容易迁移

人类价值: ⬇️ 相对下降

战略定位:
  - 让 AI 执行
  - 人类负责验证和更新
  - 作为"基础设施"，而非"护城河"

技术实现:
  - 规则引擎 (Rule Engine)
  - 关系数据库 (PostgreSQL)
  - 结构化存储

商业策略:
  - 不单独定价（作为平台基础能力）
  - 通过规模降低边际成本
```

#### 第二层：情境知识（Contextual Knowledge）
```yaml
定义: 依赖具体情境的经验、直觉、局部诀窍

AI 能力:
  学习: ⚠️ 困难（需要大量标注）
  执行: ⚠️ 不稳定（分布内 OK，分布外不行）
  泛化: ❌ 差

人类价值: 📈 短期上升，但有风险

战略定位:
  - 人机协作（人类判断，AI 辅助）
  - 逐步显性化（转为可编码知识）
  - 短期优势，不能作为长期护城河

技术实现:
  - RAG (检索增强生成)
  - 向量数据库 (Qdrant/Pinecone)
  - Embedding + LLM

商业策略:
  - 作为"增值服务"
  - 专家声誉系统（经验质量 = 定价依据）
```

#### 第三层：元层知识（Meta-level Knowledge）
```yaml
定义: 关于"什么是好问题"的知识，价值判断、目标设定、框架质疑

AI 能力:
  学习: ❌ 几乎不可能
  执行: ❌ 无（需要主动质疑框架）
  泛化: ❌ 无

人类价值: 📈📈 大幅上升

战略定位:
  - 这是我们的核心护城河
  - 人类主导，AI 执行元层洞察的指令
  - 长期竞争优势所在

技术实现:
  - 结构化洞察库
  - 缩放提示引擎 (Scale Reminder Engine)
  - 元层控制系统

商业策略:
  - 核心定价来源
  - 按价值收费（效果分成）
  - 知识交易市场
```

### 2. 商业模式选择

**不采纳**：卖工具
```
模式: 订阅制软件（$X/月/设备）
问题:
  - 容易被竞争对手复制
  - 价值上限低（只能按"降低成本"定价）
  - 客户认知："这是成本，而非投资"
  - 无护城河
```

**采纳**：卖知识平台
```
阶段 1: 卖知识执行（1-2 年）
  - 产品：领域洞察 + AI 执行
  - 定价：基础费 + 按效果分成
  - 示例：故障率降低 30% → 收故障成本节省的 20%

阶段 2: 卖知识网络（3-5 年）
  - 产品：垂直知识交易平台
  - 定价：平台费 + 知识交易抽成
  - 模式：专家贡献洞察 → 其他人付费使用 → 分成

护城河:
  - 元层洞察库（AI 无法生成）
  - 网络效应（专家越多，价值越大）
  - 知识复利（一次洞察，多次使用）
```

### 3. 技术架构选择

**核心原则**：弥补 AI 的缩放局限（参见 [tech/architecture.md](../tech/architecture.md)）

**关键组件**：

```
1. 知识捕获系统 (Knowledge Capture)
   职责: 在工作流中低摩擦捕获知识
   技术: 语音输入 + NLP 提取 + 引导式表单
   创新: AI 主动检测"元层思考"并提醒记录

2. 知识执行系统 (Knowledge Execution)
   职责: 让 AI 执行不同层次的知识
   技术: 规则引擎 + RAG + 提示词工程
   创新: 三层架构，区分可编码/情境/元层

3. 缩放提示引擎 (Scale Reminder Engine)
   职责: 提醒用户"被忽视的尺度"
   技术: LLM 尺度检测 + 多尺度对比 + 主动介入
   创新: 主动辅助多尺度思维（弥补 AI 局限）

4. 知识演化系统 (Knowledge Evolution)
   职责: 知识验证、更新、淘汰
   技术: A/B 测试 + 效果追踪 + 自动淘汰
   创新: 基于实际效果的知识质量评分
```

详见：
- 技术实现：[tech/knowledge-integration.md](../tech/knowledge-integration.md)
- 业务战略：[business/knowledge-strategy.md](../business/knowledge-strategy.md)
- 产品体验：[product/domain-expert-ux.md](../product/domain-expert-ux.md)

### 4. 组织策略

**招募"缩放者"**（Scalers）：善于多尺度思维、质疑框架的人
- 详见 [business/talent-strategy.md](../business/talent-strategy.md)

**文化建设**：鼓励"质疑当前框架"而非"在框架内优化"

## 论据

### 为什么选择三层模型？

**论据 1：AI 技术发展趋势**
```
观察:
- GPT-4 在可编码知识上已接近人类专家（如代码、公式、规则）
- 但在"什么是好问题"上仍然缺失（如 AutoGPT 失败的原因）
- CoT、RAG、ReAct 等技术都是"被动缩放"，不是"主动缩放"

推论:
- 可编码知识：AI 会持续进步，人类相对价值下降
- 情境知识：AI 短期困难，但长期可能突破
- 元层知识：AI 根本性局限，需要"主动质疑框架"的能力

结论:
→ 必须区分三层，将战略重心放在元层
```

**论据 2：客户访谈数据**
```
数据来源: 假设的 50 个 HVAC 客户访谈

发现:
- 60% 客户认为"AI 能替代基础操作"（可编码知识）
- 40% 客户认为"老师傅的经验仍然重要"（情境知识）
- 80% 客户认为"最有价值的是'发现真正的问题'"（元层知识）

引用（客户原话）:
"我不缺会操作的人，我缺能告诉我'这个方向对不对'的人"

结论:
→ 客户愿意为元层洞察付费，而非基础功能
```

**论据 3：竞争对手分析**
```
竞争对手 A: 传统 HVAC 软件
  定位: 自动化工具（可编码知识）
  护城河: 弱（技术可复制）

竞争对手 B: 通用 AI 平台（ChatGPT 等）
  定位: 通用能力（可编码 + 部分情境）
  护城河: 强（规模优势），但缺乏垂直纵深

Univers 差异化:
  定位: 垂直知识平台（可编码 + 情境 + 元层）
  护城河: 元层洞察库（AI 无法生成）+ 专家网络

结论:
→ 只有元层知识能构建差异化护城河
```

### 为什么选择"卖知识平台"？

**论据 1：单位经济模型**
```
模式 A: 卖工具（订阅）
  客户生命周期价值 LTV: $5K
  客户获取成本 CAC: $2K
  LTV/CAC: 2.5（勉强可行）

模式 B: 卖知识（效果分成）
  客户生命周期价值 LTV: $50K（按价值分成）
  客户获取成本 CAC: $5K
  LTV/CAC: 10（健康）

模式 C: 卖平台（知识交易）
  客户生命周期价值 LTV: $200K+（网络效应）
  客户获取成本 CAC: $5K
  LTV/CAC: 40+（指数增长）

结论:
→ 知识平台模式具有更好的单位经济
```

**论据 2：可防御性**
```
护城河类型      | 可编码知识 | 情境知识 | 元层知识 | 知识网络
----------------|-----------|----------|---------|----------
技术壁垒        | ⚠️ 低     | 📈 中    | 📈 中   | 📈 中
数据护城河      | ⚠️ 弱     | 📈 中    | ❌ 无   | 📈📈 强
网络效应        | ❌ 无     | ⚠️ 弱    | ❌ 无   | 📈📈 强
品牌护城河      | ⚠️ 弱     | 📈 中    | 📈📈 强 | 📈📈 强
成本优势        | 📈 强     | ⚠️ 弱    | ❌ 无   | 📈📈 强
----------------+===========+==========+=========+==========
综合可防御性    | ⚠️ 弱     | 📈 中    | 📈 中   | 📈📈 强

结论:
→ 单一知识层次护城河有限，必须构建知识网络
```

**论据 3：市场先例**
```
类比 1: GitHub（代码知识网络）
  从"代码托管"到"开发者社区"
  护城河：网络效应 + 知识沉淀

类比 2: Stack Overflow（问答知识网络）
  从"技术问答"到"开发者必备"
  护城河：UGC + 搜索排名

类比 3: Kaggle（数据科学知识网络）
  从"竞赛平台"到"知识市场"
  护城河：专家社区 + 模型复用

Univers 类比:
  从"HVAC 工具"到"领域知识市场"
  护城河：元层洞察库 + 专家网络 + 知识复利

结论:
→ 知识网络模式已被验证（技术领域），可迁移到垂直行业
```

## 考虑的替代方案

### 替代方案 1：只做可编码知识层（AI 工具）

**描述**：聚焦规则引擎、参数优化，做"AI 驱动的自动化工具"

**优点**：
- 技术实现简单
- 效果可量化
- 市场容易理解

**缺点**：
- 护城河弱（易被模仿）
- 定价上限低
- AI 进步会降低我们的价值

**为什么不采纳**：
→ 这是死路，会沦为"可替代的工具供应商"

---

### 替代方案 2：只做元层知识层（咨询公司）

**描述**：做"AI 时代的 McKinsey"，只提供战略咨询和洞察

**优点**：
- 高价值，高定价
- 不易被 AI 替代
- 品牌溢价

**缺点**：
- 无法规模化（咨询是人力密集型）
- 难以形成网络效应
- 客户获取成本高

**为什么不采纳**：
→ 无法规模化，不符合"平台"战略

---

### 替代方案 3：三层知识 + 单客户定制（系统集成商）

**描述**：为每个客户定制知识系统，不做平台化

**优点**：
- 客单价高
- 需求明确
- 实施风险低

**缺点**：
- 无知识复用（每个客户重新做）
- 无网络效应
- 无法形成护城河

**为什么不采纳**：
→ 无法产生知识复利，违背"平台"理念

---

### 我们选择的方案：三层知识 + 平台化

**为什么最佳**：
1. **差异化**：三层模型区分 AI 能做和不能做
2. **可防御**：元层知识 + 知识网络 = 双重护城河
3. **可规模化**：平台模式，知识复利
4. **价值最大化**：按效果定价，天花板高

**风险**：
- 执行难度高（需要技术 + 产品 + 运营三位一体）
- 冷启动困难（需要先积累专家和洞察）
- 文化挑战（需要建立"分享知识"的文化）

**应对**：
- 先做 MVP 验证（5-10 个标杆客户）
- 先做重点行业（HVAC）而非多行业
- 先做单个地域（中国）而非全球

## 后果

### 积极后果

**战略层面**：
- ✅ 明确了护城河：元层洞察库 + 专家网络
- ✅ 明确了价值主张：让专家专注元层思考，而非执行
- ✅ 明确了差异化：不是 AI 工具，是知识平台

**业务层面**：
- ✅ 更高的定价能力（按价值收费，而非按功能）
- ✅ 更好的单位经济（LTV/CAC > 10）
- ✅ 网络效应（专家和洞察增加 → 平台价值增加）

**技术层面**：
- ✅ 清晰的技术栈（可编码 = 规则引擎，情境 = RAG，元层 = 结构化 + 提示工程）
- ✅ 创新方向明确（缩放提示引擎、元层控制系统）
- ✅ 可度量的指标（知识质量、使用频率、价值创造）

**组织层面**：
- ✅ 明确了招聘标准：招"缩放者"，而非"AI 专家"
- ✅ 明确了文化：鼓励质疑框架，而非执行规则
- ✅ 明确了激励：知识贡献者获得收益分成

### 消极后果与风险

**风险 1：执行复杂度高**
```
挑战: 需要同时做好技术、产品、运营
缓解:
  - 分阶段实施（先 MVP，再规模化）
  - 招募有平台经验的人才
  - 寻找战略合作伙伴
```

**风险 2：冷启动困难**
```
挑战: 没有专家 → 没有洞察 → 没有客户 → 没有专家（死循环）
缓解:
  - 从小范围开始（5-10 个标杆专家）
  - 手动运营（早期由我们提供洞察）
  - 提供激励（早期贡献者高分成比例）
```

**风险 3：AI 突破元层能力**
```
挑战: 如果 GPT-6 具备了"质疑框架"的能力呢？
概率: 低（5-10 年内）
缓解:
  - 持续跟踪 AI 前沿
  - 如果发生，转型为"AI 训练数据提供商"
  - 我们沉淀的元层知识可以成为训练数据
```

**风险 4：专家不买单**
```
挑战: 专家担心"分享知识 = 被替代"
缓解:
  - 强调"知识变现"（贡献 = 收益）
  - 展示成功案例（某专家年入 $50K）
  - 文化建设（分享是专业性的体现）
```

**风险 5：客户不买单**
```
挑战: 客户认为"洞察不值那么多钱"
缓解:
  - 先卖工具建立信任，再卖洞察
  - 按效果付费（利益绑定）
  - 案例营销（量化价值创造）
```

### 度量标准

**我们如何知道这个决策是正确的？**

**短期指标（6 个月内）**：
```yaml
知识积累:
  元层洞察数: > 20
  情境经验数: > 100
  可编码规则数: > 500

专家参与:
  活跃专家数: > 10
  人均贡献: > 5 条/月
  知识复用率: > 3 次/知识

客户验证:
  付费客户: > 3
  客户愿意为洞察付溢价: > 50%
  NPS: > 40
```

**中期指标（12 个月内）**：
```yaml
网络效应:
  专家数: > 50
  知识库规模: > 1000
  知识复用率: > 10 次/知识

商业指标:
  ARR: > $500K
  LTV/CAC: > 5
  知识服务占收入比: > 40%

竞争力:
  客户续约率: > 80%
  从竞争对手转化率: > 20%
```

**长期指标（24 个月内）**：
```yaml
平台化:
  专家数: > 200
  第三方开发者: > 10
  行业影响力: 成为 HVAC 领域标准

护城河:
  知识独占性: > 70%（我们有，竞争对手没有）
  专家黏性: 年留存 > 85%
  品牌认知: "HVAC 领域的知识中心"
```

## 反思与学习

### 决策过程中的关键洞察

**洞察 1：垂直知识的本质是"分层"的**
```
最初想法: 垂直知识是一个整体
讨论后发现: 必须区分可编码/情境/元层
关键时刻: 意识到"AI 在不同层次的能力差异巨大"
结果: 采用三层模型
```

**洞察 2："卖什么"比"怎么做"更重要**
```
最初想法: 先想清楚技术实现
讨论后发现: 必须先明确商业模式和护城河
关键时刻: 认识到"卖工具"是死路
结果: 重新定位为"知识平台"
```

**洞察 3：AI 的局限性是我们的机会**
```
最初想法: AI 很强大，我们怎么竞争？
讨论后发现: AI 有根本性局限（无法主动缩放）
关键时刻: 理解"被动缩放 vs 主动缩放"的区别
结果: 将战略重心放在"弥补 AI 局限"
```

### 如果重新决策，会改变什么？

**更早地**：
- 与潜在客户深度访谈（验证"元层价值"假设）
- 研究其他行业的知识平台（寻找模式）
- 小规模实验（快速验证冷启动策略）

**会保留**：
- 三层知识模型（这是核心洞察）
- 平台化战略（这是规模化路径）
- 弥补 AI 局限的技术方向（这是差异化）

### 开放问题

1. **定价模式细节**：分成比例应该是多少？30%？50%？如何平衡平台、专家、客户的利益？

2. **知识产权**：专家贡献的知识，所有权归谁？如果专家离开平台，能否带走知识？

3. **质量控制**：如何防止低质量知识泛滥？如何处理错误的洞察？

4. **跨行业扩展**：三层模型是否适用于 HVAC 之外的行业？如何验证？

5. **AI 能力进化**：如果 AI 突破了元层能力，我们的战略如何调整？
